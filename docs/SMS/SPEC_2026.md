# SMS Language Specification 2026

Status: Draft (normative target for 2026)
Audience: SMS language authors, runtime implementers, tooling authors

## 1. Purpose

This document defines the normative SMS 2026 language surface and runtime semantics.
It is designed as a compatibility-first evolution of current SMS with a strict focus on:

- UI scripting ergonomics
- deterministic behavior across managed and native runtimes
- minimal language complexity

Design principle: built with love, coffee, and a stubborn focus on simplicity.

## 2. Compatibility Model

SMS 2026 is a delta-compatible update.

- Existing scripts should continue to run unless they rely on behavior that is now explicitly rejected.
- When behavior changes are required, runtimes must produce explicit diagnostics.
- No silent semantic fallback is allowed for newly specified errors.

## 3. Normative Features

### 3.1 Modules: `import` and `export`

#### 3.1.1 Syntax

```sms
import "res:/libs/math.sms"
import "appRes:/modules/ui.sms" as ui

export fun clamp(v, min, max) { ... }
export var defaultTheme = "light"
```

#### 3.1.2 Rules

- Import resolution is scheme-rooted and deterministic.
- Module resolution must be deterministic and independent of host OS search order.
- Allowed import path forms are:
  - `res:/...`
  - `appRes:/...`
  - optional runtime-local `user:/...` (if enabled by host policy)
- Relative imports are forbidden (`./...`, `../...` are invalid).
- `//` is forbidden in import paths.
  - `res://...`, `appRes://...`, `user://...`, `http://...`, `https://...`, and `file://...` are invalid in `import`.
- Only explicitly exported symbols are importable from a module namespace.
- Import cycles must be diagnosed with a clear cycle path.
- Duplicate exported symbol names in one module are an error.

### 3.2 Optional Typing Surface

#### 3.2.1 Syntax

```sms
var retries: Int32 = 3
fun sum(a: Int32, b: Int32): Int32 { return a + b }
```

#### 3.2.2 Rules

- Type annotations are optional on variables, parameters, and function returns.
- Missing annotations keep existing dynamic behavior.
- Type annotations are normative for transpiler/type-check stages.
- Runtime interpreters may enforce type checks eagerly or lazily, but diagnostics must be consistent in meaning.

### 3.3 Named Arguments

#### 3.3.1 Syntax

```sms
drawRect(100, 50, color = "red", filled = true)
Vec3D(y = 100)
```

#### 3.3.2 Binding Rules

- Positional arguments may appear first.
- After the first named argument, all remaining arguments must be named.
- Unknown named argument is an error.
- Duplicate named argument is an error.
- Omitted parameter uses default value when defined; otherwise it is an error.
- Named argument order is irrelevant.

### 3.4 Collection Loop: `for (item in list)`

#### 3.4.1 Syntax

```sms
for (item in items) {
    log.info(item)
}
```

#### 3.4.2 Rules

- The right-hand expression is evaluated once before iteration.
- Iteration order follows collection index order for arrays/lists.
- Non-iterable values are a runtime error with source position.
- Loop variable scope is the loop body scope.

### 3.5 Non-Blocking Callback Model

#### 3.5.1 Scope

SMS 2026 does not include language-level async/await.

- `await` is not part of this version.
- User-declared `async fun` is not part of this version.
- Non-blocking behavior is provided via runtime APIs with callback contracts.

#### 3.5.2 Non-Blocking Callback Contract

For runtime operations that may block (for example network and I/O), APIs must use callback-style completion.

- `onSuccess`: required callback target, invoked on successful completion.
- `onError`: required callback target, invoked on failure.
- `timeoutMs`: optional timeout in milliseconds; if omitted, default is `5000`.

Callback targets must be symbol references (for example `onSuccess = myCallback`) and not string names.

#### 3.5.3 UI Responsiveness Rules

- Any operation expected to exceed 50 ms must not block the UI thread.
- Network and I/O operations must be non-blocking from UI script execution paths.
- Callback dispatch for UI-observable effects must run on the main thread.

#### 3.5.4 Error Rules

- Failures must never be silently swallowed.
- Unhandled callback failures must be reported with script location and handler/function context.
- On unhandled failure, the active handler/function must abort in a controlled way.

#### 3.5.5 Runtime API Visibility

The runtime must expose at least one explicit background-task entrypoint to scripts.

Minimum required capability:

```sms
bg.run(taskName, data, onSuccess = callbackA, onError = callbackB, timeoutMs = 5000)
```

Contract requirements:

- `taskName`: identifies a runtime-provided background operation.
- `data`: optional task payload.
- `onSuccess`: required callback symbol.
- `onError`: required callback symbol.
- `timeoutMs`: optional; default is `5000` when omitted.
- `timeoutMs <= 0` is invalid and must fail fast with a binding/runtime diagnostic.
- `bg.run(...)` returns a `requestId` value for correlation and optional cancellation.
  - `requestId` must be a runtime-generated UUID string in canonical lowercase hyphenated form (`xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`).
- Callback signatures are fixed:
  - `onSuccess(result)`
  - `onError(error)`

The exact module/object naming can vary by host runtime, but an equivalent explicit API must exist.

#### 3.5.6 Cross-Thread Data Contract

Data exchange between main-thread script execution and background tasks is value-based only.

- Input (`data`) and outputs (`result`, `error`) must be SML-serializable value structures.
- DTO-style payloads are allowed, including SMLNode-like snapshot objects as pure values.
- Shared mutable objects, runtime handles, UI object references, and closures must not cross the thread boundary.
- Transfer semantics are copy/snapshot semantics at the contract level.

Serialization policy by backend:

- Interpreter/runtime path: SML value serialization is the canonical contract format.
- Transpiled backend path: binary representation is allowed, but it must preserve the same value contract semantics.

#### 3.5.7 Result/Error Shape and Correlation

Callback payloads must use object-shaped DTO values.

- `result` minimum fields:
  - `requestId`
  - `data` (task-specific payload)
- `error` minimum fields:
  - `requestId`
  - `code`
  - `message`
  - optional `details`

`requestId` in callback payload must match the `requestId` returned by `bg.run(...)`.
The callback `requestId` value must preserve the exact UUID string returned by `bg.run(...)`.

`error.code` is an enum-like string with a fixed set of allowed values:

- `Timeout`
- `Cancelled`
- `NotFound`
- `AccessDenied`
- `InvalidInput`
- `InvalidState`
- `Conflict`
- `RateLimited`
- `Unavailable`
- `Internal`

Unknown error codes are not allowed in conforming runtimes.

#### 3.5.8 Ordering and Cancellation

- Callback dispatch order is guaranteed per `requestId` only.
- No global ordering is guaranteed across different requests.
- Runtime must expose cancellation for inflight background tasks:
  - minimum capability: `bg.cancel(requestId) -> bool`
  - return value is deterministic:
    - `true`: cancel signal accepted for a known non-terminal request
    - `false`: unknown `requestId` or request already terminal
  - `bg.cancel` is idempotent; repeated calls after terminal state return `false`.
  - cancellation is non-blocking and must not run task cleanup on the UI thread.

Terminal-state rule (anti-race contract):

- Each `requestId` has exactly one terminal state: `Success`, `Error`, `Cancelled`, or `Timeout`.
- Terminal transition is atomic. First terminal transition wins.
- Exactly one terminal callback is dispatched per `requestId`:
  - `onSuccess(result)` for `Success`
  - `onError(error)` for `Error`, `Cancelled`, or `Timeout`
- A canceled request must never dispatch `onSuccess`.

## 4. Threading and Safety Guarantees

- Script-visible UI state is main-thread owned.
- Background tasks must not mutate UI state directly.
- Managed and native runtimes must expose equivalent safety behavior at SMS level.

## 5. Determinism and Parity Requirements

- Managed and native runtimes must pass the same conformance suite for all features in this spec.
- Parser and semantic diagnostics should match in category and intent, even if wording differs.
- Identical SMS input should produce deterministic parse and execution behavior under the same runtime configuration.

## 6. Diagnostics Requirements

Diagnostics for parse, bind, and runtime errors must include:

- message
- line and column
- script/module identifier
- error category (parse, binding, runtime, task)

## 7. Conformance Examples

### 7.1 Module + Named Arguments

```sms
import "appRes:/modules/vec.sms" as vec

fun demo() {
    var v = vec.Vec3D(y = 10)
    log.info(v.y)
}
```

### 7.2 `for-in` over array

```sms
var values = [1, 2, 3]
var sum = 0

for (v in values) {
    sum = sum + v
}
```

### 7.3 Non-blocking network call

```sms
on saveButton.clicked() {
    var requestId = http.get(
        "/api/project",
        onSuccess = projectLoaded,
        onError = projectLoadFailed,
        timeoutMs = 5000
    )
    log.debug("request started: ${requestId}")
}

on projectLoaded(result) {
    ui.setText("statusLabel", result.data.status)
}

on projectLoadFailed(error) {
    log.error(error.message)
}
```

### 7.4 Background task call

```sms
on exportButton.clicked() {
    bg.run(
        "ExportGlb",
        { path: "user:/exports/out.glb" },
        onSuccess = exportDone,
        onError = exportFailed,
        timeoutMs = 10000
    )
}

on exportDone(result) {
    ui.setText("statusLabel", "done: " + result.requestId)
}

on exportFailed(error) {
    log.error(error.message)
}
```

## 8. Non-Goals (2026)

- Full coroutine/task authoring model in SMS syntax
- User-defined `async fun`
- Language-level `await`
- Full SMS to C++ compiler definition in this document
